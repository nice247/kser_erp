# نظام إدارة غرف الطوارئ - KSER ERP (Odoo 18 Module)

نظام متكامل لإدارة العمليات الميدانية، الطبية، المالية، والمخازن الإغاثية لغرف الطوارئ في السودان (ولاية الخرطوم)، مصمم ومبني خصيصاً على منصة Odoo 18 ومهيأ بالكامل للغة العربية والاتجاه من اليمين لليسار (RTL).

يدعم النظام حوكمة صارمة لمنع التلاعب وتكرار الصرف، ويعمل كنظام ويب تقدمي (PWA) مع قاعدة بيانات محلية (IndexedDB) للعمل دون اتصال بالإنترنت في الميدان.

---

## 🛠️ المميزات الرئيسية وخصائص المشروع

### 1. إدارة المستفيدين والتحقق الذكي (Beneficiaries)
* **تسجيل ذكي بالـ OCR**: استخراج بيانات الهوية الوطنية (الرقم الوطني، الاسم، العمر، إلخ) تلقائياً عبر فحص صورة الهوية باستخدام نماذج الذكاء الاصطناعي (مثل Gemini API أو أي مزود خدمة ذكاء اصطناعي آخر يدعم الرؤية البصرية، بما في ذلك النماذج المحلية).
* **التحقق الآلي من العلاقات الأسرية**: دعم هيكلي لتحديد رب الأسرة وربط بقية أفراد العائلة كتابعين، مع حجم عائلة ديناميكي وتوثيق الفئات العمرية.
* **إلزامية الهويات**: منع تسجيل أي مستفيد دون التحقق من صحة الرقم الوطني لمنع التسجيل المزدوج والكشوفات الوهمية.

### 2. منظومة العيادة الطبية اليومية (Clinic Management)
* **إدارة طابور العيادة**: تسجيل زيارات المرضى وتوجيههم للأطباء ضمن طابور انتظار يومي مرقم آلياً (VS-XXXXX).
* **الروشتات والصرف المخزني**: كتابة الروشتات الطبية إلكترونياً، وعند قيام الصيدلي بصرفها، يُنشئ النظام قيد صرف مخزني (`stock.picking`) لخصم كميات الأدوية تلقائياً.
* **متابعة سوء التغذية للأطفال**: تسجيل القياسات الجسدية (طول، وزن) للأطفال أقل من 7 سنوات، مراقبة تطور الأوزان، وتحديد مواعيد الزيارات القادمة.

### 3. إدارة التبرعات النقدية والعينية
* **التبرعات النقدية الذكية**: معالج ذكي يقرأ إشعارات التحويل البنكي (مثل تطبيق بنكك أو فوري) ويستخرج بيانات المعاملة للتحقق من عدم تكرارها، مسجلاً التبرع مع توليد سندات القبض وقيود اليومية المحاسبية فور التأكيد.
* **التبرعات العينية**: استلام المواد بالمخزن مع تحديد تاريخ الصلاحية ورقم الدفعة واستخدام آلية FEFO لصرف الدفعات الأقرب لانتهاء الصلاحية.

### 4. محرك تحليل وتوزيع المخزون (Inventory Gap Analysis)
* **المطابقة الذكية للاحتياج**: محرك داخلي يقوم بتحليل أرصدة الأدوية المتاحة (المواد الفعالة) ومطابقتها مع الأمراض المزمنة المسجلة للمستفيدين لاقتراح قائمة توزيع مثالية تسد الفجوات وتولد أذونات الصرف تلقائياً لتغطية العجز.

### 5. إدارة المصروفات وحوافز المتطوعين (Cash Expenses)
* **المصروفات والحوافز للمشرف الإداري**: نظام متكامل لإدارة المصروفات التشغيلية وحوافز المتطوعين يقع هيكلياً تحت قائمة **"المهام والحملات"** ومخصص حصرياً لـ **المشرف الإداري** و**مسؤول النظام**.
* **الربط المحاسبي التلقائي**: عند تأكيد واعتماد المصروف، يتم توليد قيد محاسبي ترحيلي (`account.move`) وربطه تلقائياً بـ **"دفتر البنك"** الرئيسي وحساب **البنك الرئيسي** (`11002`) لضمان انخفاض رصيد الخزينة والبنك بالفوترة بشكل دقيق وظهور المعاملة بالكامل تحت كشوفات البنك.
* **الفلترة الديناميكية الذكية للمتطوعين**: لا يمكن اختيار متطوع لصرف حافز إلا إذا كان لديه **مهمة موكلة إليه ومسندة بالفعل داخل نفس الحملة المعنية** لمنع الأخطاء البشرية أو الصرف العشوائي لغير المستحقين.
* **حوكمة الصرف المشروطة**: قاعدة رقابية برمجية تمنع ترحيل أي قيد مالي لصرف حافز متطوع ما لم يملك مهمة واحدة على الأقل بحالة **"مكتملة"** (`state = '1_done'` أو مرحلة مطوية أو نسبة إنجاز %100) في الحملة المستهدفة.

### 6. الحوكمة والأمان والتتبع
* **سجل التدقيق (Audit Log)**: شاشة مركزية ترصد كافة العمليات الحساسة (إنشاء، تعديل، اعتماد) وتوثق التغييرات في سجل غير قابل للحذف لضمان أعلى مستويات الشفافية.
* **لوحة القيادة المخصصة والفلترة المضمونة**: لوحة قيادة مركزية تعرض أهم مؤشرات الأداء وجدولاً بأحدث جهات الاتصال المسجلة، مهيأة برمجياً لاستبعاد أي "مستخدم داخلي" للنظام تلقائياً باستخدام حقول المهام (Role-Based Boolean Fields) لضمان نظافة ونقاء البيانات المعروضة.
* **صلاحيات متعددة المستويات**: نظام تحكم بالوصول المبني على الأدوار (RBAC) يخصص الصلاحيات (مشرف ميداني، مسؤول مخازن، مسؤول بيانات، محاسب، صيدلي، مشرف إداري، وطبيب).

### 7. التقارير الاستراتيجية والتشغيلية (PDF)
توليد تقارير لحظية بصيغة QWeb وتصديرها كـ PDF بضغطة زر:
* **تقرير أداء الغرفة وحجم المساعدات**.
* **تقرير إنجاز المتطوعين الميدانيين**.
* **تقرير الميزانية التشغيلية**.
* **تقرير أداء العيادات الطبية**.

### 8. الربط الهجين ودعم الأوفلاين (Offline PWA)
* **معمارية Offline-First**: يعتمد على `Service Worker` لاعتراض طلبات الشبكة وحفظ بيانات المعاملات مؤقتاً ككائنات JSON في قاعدة `IndexedDB` المحلية عند انقطاع الإنترنت.
* **المزامنة الخلفية التلقائية**: بمجرد استشعار عودة الاتصال (Sync Event)، تُرفع البيانات المؤقتة آلياً لخادم أودو وتُمحى من المتصفح لمنع الازدواجية.

### 9. بيئة تطوير معرّبة بالكامل (Arabic Developer Environment)
* **تعريب الكود المصدري**: تم تعريب كافة التعليقات التوضيحية والتوثيقية داخل الشيفرة البرمجية (ملفات Python وJavaScript وXML) بالكامل لدعم المطورين والمساهمين المحليين وتسهيل قراءة الكود وصيانته.

## ⚙️ آليات العمل الفنية (Technical Mechanisms)
* **التكامل مع الذكاء الاصطناعي**: دمج مرن عبر مكتبة `requests` للاتصال بأي مزود خدمة API لنموذج ذكاء اصطناعي يدعم قراءة الصور (مثل `Google Gemini API` أو أي خادم ذكاء اصطناعي محلي) لتنفيذ مهام الـ OCR الذكية للوثائق والإيصالات البنكية.
* **معالجة الصور**: يتم ضغط وتحسين الصور محلياً قبل إرسالها لتقليل حجم الحمولة (Payload) باستخدام مكتبة بايثون `Pillow (PIL)`.
* **محاكاة استجابة الـ Odoo RPC**: أثناء انقطاع الشبكة، يُصدر الـ Service Worker استجابة وهمية (Mock Deferred Response) لمحاكاة رد الخادم، مما يحول دون ظهور رسائل الخطأ أو انهيار واجهة النظام ويسمح للمتطوع بمواصلة إدخال الاستمارات بسلاسة تامة.

## 📂 بنية مجلدات الموديول
* `models/`: يحتوي على منطق الحوكمة ونماذج البيانات (المستفيدين، التبرعات، المصروفات، العيادة، المتابعة، سجل التدقيق).
* `views/`: تصاميم واجهات المستخدم (List, Form, Kanban, Search) وتخصيص القوائم والمجلدات المالية المدمجة.
* `wizard/`: النوافذ المنبثقة لمعالجات التحقق الذكي للـ OCR والتبرعات وتشخيصات المخزون المالي والعيني.
* `security/`: تعريف أدوار النظام وقواعد الوصول وجداول الصلاحيات (`ir.model.access.csv`).
* `report/`: تقارير QWeb الجاهزة للطباعة وقوالب التصدير.
* `static/`: الأصول البرمجية، تصاميم SCSS المتجاوبة الخاصة بالموديول، وملفات الجافاسكريبت لخدمة الـ Offline-First والمزامنة.
* `data/`: قواميس الربط الطبي (الأمراض والمواد الفعالة) وبيانات تهيئة النظام الافتراضية.

## 🚀 التثبيت والتهيئة (Installation & Setup)

### المتطلبات مسبقة الصنع:
* بيئة Odoo 18.0 مثبتة وجاهزة للعمل.
* مكتبات Python الأساسية: `requests`, `Pillow`.
* الحصول على مفتاح API نشط لمزود خدمة ذكاء اصطناعي يدعم قراءة الصور (مثل `Google Gemini AI` أو بديل محلي متوافق).

### خطوات التثبيت:
1. انسخ مجلد الموديول `kser_erp` بالكامل إلى مسار الـ `addons` الخاص بـ Odoo.
2. تأكد من توفر الاعتماديات المطلوبة للموديول كما هي معرّفة في ملف `__manifest__.py`:
   `['base', 'mail', 'project', 'stock', 'product', 'account', 'web', 'board', 'contacts', 'spreadsheet_dashboard', 'web_pwa_customize']`
3. قم بتحديث قائمة التطبيقات في أودو:
   * قم بتفعيل **وضع المطور (Developer Mode)**.
   * انتقل إلى **التطبيقات (Apps)** -> **تحديث قائمة التطبيقات (Update Apps List)**.
4. ابحث عن `KSER ERP` واضغط على **تثبيت (Install)**.
5. **إعداد الذكاء الاصطناعي:** بعد نجاح التثبيت، انتقل إلى الإعدادات > إعدادات KSER ERP (أو إعدادات النظام العامة) وقم بإدخال مفتاح الـ API الخاص بمزود خدمة الذكاء الاصطناعي لتفعيل قدرات الـ OCR والمطابقات الذكية للنظام.
